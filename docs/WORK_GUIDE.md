# 项目工作指南

## A. 项目特征记录

### 1. 目录结构规范
- 根目录：功能性PHP和HTML文件
- /assets：前端资源文件
  - /css：样式文件
  - /js：脚本文件
- /docs：开发文档目录
- /uploads：上传文件目录

### 2. 文档规范
- 主文档：README.md（小写）
- 开发日志：docs/DEV_LOG.md（大写）
- 文档风格：简洁明了，层级清晰

### 3. 文件命名规范
- 页面文件：全小写（index.html, admin.html）
- PHP文件：全小写（api.php, config.php）
- 资源文件：全小写（common.css, common.js）
- 第三方库：保持原名（bootstrap.min.css）

### 4. 代码修改原则
- 严格遵循最小修改原则
- 只修改指定功能相关的代码
- 保持其他功能代码不变
- 如需修改其他代码，必须先获得确认
- 每次修改前明确影响范围

### 5. 开发日志规范
- 重要功能更新必须记录
- 重大bug修复必须记录
- 架构调整必须记录
- 细节性的微调（如小幅度的样式调整）可以不记录
- 日志要关注对功能和用户体验有实质影响的改动

### 6. 特殊注意点
- 不使用大写文件名（除了文档）
- 使用下划线分隔词（dev_log）
- 保持目录结构扁平化

## B. 工作流程检查表

### 1. 接收任务前
□ 完整阅读需求描述
□ 标记关键词和重点
□ 确认是否理解全部内容
□ 如有疑问立即提出

### 2. 开始工作前
□ 运行项目环境自检清单
bash
获取目录结构
ls -R > project_structure.temp
查找所有文档
find . -name ".md"
检查文件命名规范

## 检查代码组织方式

### 3. 编码过程中
□ 随时对照需求文档
□ 只关注相关代码块
□ 在修改前添加注释标记
□ 保存修改前的代码备份

### 4. 提交更改前
□ 检查是否有多余的修改
□ 确认所有更改都是必要的
□ 验证功能是否正常
□ 检查文档是否需要更新
□ Git 提交前检查：
  - □ 确认当前分支是否正确
  - □ 检查待提交的文件列表（git status）
  - □ 查看具体更改内容（git diff）
  - □ 确认提交信息符合规范
  - □ 确认是否需要更新开发日志
  - □ 确认是否需要更新 README.md

### 5. 完成工作后
□ 对照原始需求复查
□ 确认没有遗漏任何步骤
□ 整理并提交相关文档
□ 记录经验和教训
□ Git 合并检查：
  - □ 确认功能分支所有改动已提交
  - □ 确认主分支是最新版本
  - □ 进行代码冲突检查
  - □ 确认合并后的代码能正常运行
  - □ 合并完成后删除功能分支

## C. 使用说明

1. 每次开始新任务时：
   - 先阅读项目特征记录
   - 对照工作流程检查表
   - 严格执行每个检查步骤

2. 持续改进：
   - 记录遇到的新问题
   - 更新检查项内容
   - 完善工作流程

3. 定期复查：
   - 每周回顾工作记录
   - 总结经验教训
   - 优化工作方法

## D. Git 工作流程

### 1. 分支管理
- 主分支：main
- 当前使用的功能分支：
  - feature/page-split（页面拆分重构）
- 功能分支命名规范：
  - 新功能：feature/[功能名称]
  - 修复问题：fix/[问题描述]
  - 文档更新：docs/[文档描述]

### 2. 提交流程
□ 创建功能分支
  ```bash
  git checkout -b feature/[功能名称]
  ```

□ 提交到功能分支
  ```bash
  git add [文件]
  git commit -m "type: 提交描述"
  git push origin feature/[功能名称]
  ```

□ 确认无误后合并到主分支
  ```bash
  git checkout main
  git pull origin main
  git merge feature/[功能名称]
  git push origin main
  ```
□ 合并到主分支后删除功能分支
  ```bash
  git branch -d feature/[功能名称]
  ```

### 3. 提交信息规范
- feat: 新功能
- fix: 修复问题
- docs: 文档更新
- style: 代码格式调整
- refactor: 代码重构
- test: 测试相关
- chore: 构建过程或辅助工具的变动

### 4. 注意事项
- 每次提交前先拉取最新代码
- 确保功能分支测试通过
- 重要更改需要代码审查
- 合并到main前必须确认无误
